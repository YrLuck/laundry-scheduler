# Отчёт по лабораторным работам — Fullstack Laundry Scheduler

## Что за проект

**Laundry Scheduler** — веб-приложение для бронирования стиральных машин и сушилок.  
Пользователи регистрируются, выбирают свободный временной слот и бронируют машину.  
Администратор управляет оборудованием и пользователями.

### Стек технологий
| Слой | Технология |
|---|---|
| Backend | FastAPI (Python 3.11) |
| База данных | SQLite (через SQLAlchemy ORM) |
| Аутентификация | JWT (access + refresh токены, библиотека python-jose) |
| Хеширование паролей | bcrypt |
| Хранилище файлов | MinIO (S3-совместимое) |
| Frontend | React (TypeScript) |
| Деплой | Docker Compose |

## Лабораторная работа 1 — Архитектура и модели данных

### Что реализовано

Проект построен по **трёхслойной архитектуре**:

```
HTTP Request
    ↓
Router (routers/)       ← принимает запрос, валидирует через Pydantic схемы
    ↓
CRUD / Service layer    ← бизнес-логика, работа с данными
    ↓
Repository / Models     ← SQLAlchemy ORM, SQLite БД
```

### Модели базы данных (`app/models.py`)

**User** — пользователь системы:
- `username`, `email`, `full_name` — основные поля
- `hashed_password` — пароль хранится в виде bcrypt-хеша (никогда в открытом виде)
- `role_id` — внешний ключ на таблицу ролей
- `is_active` — флаг блокировки аккаунта

**Machine** — стиральная машина или сушилка:
- `type` — `"washer"` или `"dryer"`
- `status` — `"available"` / `"in_use"` / `"maintenance"`

**Booking** — бронирование:
- `machine_id`, `user_id` — внешние ключи
- `start_time`, `end_time` — временной слот
- `status` — `"active"` / `"completed"` / `"cancelled"`
- `user_name` — денормализованное поле для быстрого отображения без JOIN

**Role / Permission** — RBAC-система:
- Таблица `role_permissions` связывает роли и разрешения (many-to-many)


> "Мы спроектировали реляционную базу данных с пятью таблицами. Ключевое решение — использование паттерна RBAC (Role-Based Access Control) через связку Role → Permission через ассоциативную таблицу. Это позволяет гибко управлять правами без изменения кода."

## Лабораторная работа 2 — Аутентификация и JWT-токены

### Что реализовано

Система использует **двухтокенную схему авторизации** (Access + Refresh):

```
POST /auth/register  →  создание аккаунта, пароль хешируется bcrypt
POST /auth/login     →  проверка пароля → выдача двух JWT-токенов
GET  /auth/me        →  получить данные о себе (нужен access token)
POST /auth/refresh   →  обновить access token через refresh token
POST /auth/logout    →  клиент удаляет токены (stateless logout)
```

### Как работают токены

**Access token** — короткоживущий (30 минут). Передаётся в каждом запросе в заголовке:
```
Authorization: Bearer <access_token>
```

**Refresh token** — долгоживущий (7 дней). Используется только для получения нового access token.  
Оба токена содержат поле `"type"` — защита от использования refresh token вместо access.

### Структура JWT payload
```json
{
  "sub": "testuser",
  "exp": 1778850601,
  "type": "access"
}
```

### Пример: создание тестового пользователя
```bash
# Регистрация
curl -X POST http://localhost:8000/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username": "testuser", "email": "test@example.com", "password": "Pass123"}'

# Вход — получаем оба токена
curl -X POST http://localhost:8000/auth/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=testuser&password=Pass123"
```

**Результат:**
```json
{
  "access_token": "eyJhbGci...",
  "refresh_token": "eyJhbGci...",
  "token_type": "bearer"
}
```


> "Мы реализовали stateless-аутентификацию через JWT. При логине сервер выдаёт пару токенов: короткоживущий access (30 мин) и долгоживущий refresh (7 дней). При истечении access токена клиент отправляет refresh на `/auth/refresh` и получает новую пару без повторного ввода пароля. Пароли никогда не хранятся в открытом виде — только bcrypt-хеш."

## Лабораторная работа 3 — RBAC (Role-Based Access Control) это мой гриб я его ем 

### Что реализовано

Три предустановленные роли с разными наборами прав:

| Роль | Разрешения |
|---|---|
| `guest` | только просмотр (нет разрешений в системе) |
| `user` | `bookings:create`, `bookings:update` |
| `admin` | все 8 разрешений |

**8 разрешений системы:**
- `machines:create`, `machines:update`, `machines:delete`
- `bookings:create`, `bookings:update`, `bookings:delete`
- `users:manage`, `roles:manage`

### API RBAC-системы
```
GET  /rbac/roles/                          — список ролей (публично)
GET  /rbac/permissions/                    — список разрешений (публично)
POST /rbac/users/{user_id}/assign-role/{role_id}  — назначить роль (только admin)
GET  /rbac/users/me/permissions            — мои разрешения (нужен токен)
```

### Как проверяются права в коде
```python
# В роутере — проверка разрешения:
@router.post("/machines/")
def create_machine(
    machine: MachineCreate,
    current_user = Depends(auth.require_permission("machines:create")),
    ...
):
```


> "RBAC реализован через dependency injection FastAPI. Функции `require_role()` и `require_permission()` — это фабрики зависимостей. Они возвращают async-функцию, которую FastAPI автоматически вызывает перед роутером. Admin всегда проходит проверку без исключений — это сделано специально, чтобы не дублировать логику."

## Лабораторная работа 4 — Бронирования и бизнес-логика

### Что реализовано

```
POST /bookings/                    — создать бронирование
GET  /bookings/my-bookings         — мои бронирования
GET  /bookings/user/{user_id}      — бронирования пользователя
POST /bookings/{id}/cancel         — отменить бронирование
PUT  /bookings/{id}                — изменить бронирование
GET  /machines/{id}/time-slots     — свободные слоты машины на дату
```

### Ключевая бизнес-логика — проверка конфликтов

При создании бронирования система проверяет все три варианта пересечения временных слотов:

```python
or_(
    and_(Booking.start_time <= new.start_time, Booking.end_time > new.start_time),
    and_(Booking.start_time < new.end_time,    Booking.end_time >= new.end_time),
    and_(Booking.start_time >= new.start_time, Booking.end_time <= new.end_time)
)
```

Это покрывает случаи: новый слот **внутри** существующего, **вокруг** него, или **частично пересекается**.

### Фильтрация и пагинация

Все списковые endpoint'ы поддерживают:
- `skip` / `limit` — пагинация
- `status`, `user_id`, `machine_id` — фильтрация
- `sort_by`, `sort_order` — сортировка


> "Самая сложная часть — алгоритм проверки пересечений времени. Мы используем три условия OR, которые покрывают все возможные варианты наложения двух временных отрезков. Если конфликт найден — возвращаем ошибку 400. Временные слоты генерируются на стороне сервера — с 8:00 до 22:00 по часу."

## Лабораторная работа 5 — S3 и работа с файлами

### Что реализовано

Интеграция с **MinIO** — S3-совместимым хранилищем, которое разворачивается в Docker.

```
POST /files/upload              — загрузка файла (max 10MB)
GET  /files/{object_name}       — скачивание файла
GET  /files/{object_name}/url   — получить временную ссылку (presigned URL)
DELETE /files/{object_name}     — удалить файл
GET  /files/                    — список файлов пользователя
```

### Организация хранилища

Файлы хранятся с префиксом пути: `user_{id}/YYYY/MM/DD/{uuid}.ext`  
Это позволяет:
1. Изолировать файлы каждого пользователя
2. Проверять права доступа по префиксу пути
3. Логически группировать файлы по дате загрузки

### Допустимые типы файлов
`image/jpeg`, `image/png`, `image/gif`, `application/pdf`, `text/plain`, `application/json`

> "Мы использовали MinIO как S3-совместимое хранилище — это позволяет в production заменить его на AWS S3 без изменения кода (тот же boto3/S3 API). Presigned URL позволяет давать временный доступ к файлу напрямую из браузера, не проксируя трафик через backend."

## Лабораторная работа 6 — Внешние API и SEO

### Внешний API — OpenWeatherMap

```
GET /external/weather?lat=55.75&lon=37.61        — текущая погода
GET /external/weather/forecast?lat=...&lon=...   — прогноз на 1-5 дней
GET /external/weather/recommendation?lat=...     — рекомендация: можно ли сушить на улице?
```

Если `OPENWEATHER_API_KEY` не задан — возвращаются mock-данные (без падения сервиса).

### SEO endpoints

```
GET /robots.txt            — инструкции для поисковых роботов
GET /sitemap.xml           — карта сайта для индексации
GET /manifest.json         — PWA-манифест
GET /.well-known/security.txt  — политика безопасности
GET /api/schema.json       — JSON-LD Schema.org разметка
```


> "SEO и манифест реализованы на backend — это позволяет поисковым ботам получить robots.txt и sitemap прямо от API. Интеграция с погодным API добавляет реальную пользу: рекомендация помогает пользователям решить, стоит ли вешать бельё на улицу или лучше воспользоваться сушилкой."
